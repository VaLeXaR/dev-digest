## What Doesn't Work

- 2026-06-22: For LLM extraction tasks that must return `[]` on parse errors (not throw), use `llm.complete()` — NOT `llm.completeStructured()`. `completeStructured()` throws a validation error when the model response doesn't match the schema, which breaks the `[]`-on-error contract. `complete()` returns raw text and lets you catch `JSON.parse` failures yourself. (`src/modules/conventions/extractor.ts:callLLM`)

- 2026-06-20: `cost_usd` was deliberately removed from `agent_runs` in migration `0009_complex_runaways.sql` (`ALTER TABLE "agent_runs" DROP COLUMN "cost_usd"`). It was re-added in `0010_grey_naoko.sql`. If you see the column missing and old data showing "—", that is expected — only runs after migration 0010 have persisted cost.

- 2026-06-23: `buildSamples` searches for config files (`tsconfig.json`, `.eslintrc.*`, etc.) only at the repo root — it never descends into subdirectories. Monorepos (DevDigest: `server/tsconfig.json`, `client/tsconfig.json`) get **0 config samples**. JavaScript SPAs with root-level configs get 1–2 at best. With an unindexed repo (no repo-intel), the LLM receives a near-empty input and returns `[]`. Fix: glob the config files recursively or add subdirectory paths to the search list. (`src/modules/conventions/extractor.ts:27-96`)

- 2026-06-23: The conventions system prompt contains "Do NOT include generic best practices obvious to any TypeScript developer" — for JavaScript repos, the LLM treats most Prettier/ESLint conventions as "obvious JS knowledge" and discards them, silently producing `[]`. The prompt is implicitly TypeScript-only. (`src/modules/conventions/extractor.ts:100`)

- 2026-06-26: `IntentService.generate()` initially called `this.container.db.select().from(t.pullRequests)` directly — raw Drizzle queries from a service class bypass the repository layer (onion invariant: service → repository → db). Architecture reviewer flagged HIGH. Fix: construct `ReviewRepository(container.db)` in the constructor, call `this.repo.getPull()`, `getPrFiles()`, `getRepo()` instead, and remove the `* as t` and bare `drizzle-orm` imports. (`src/modules/intent/service.ts`)

- 2026-07-02: A `sort().slice(0, N)` applied AFTER merging callers from multiple changed symbols into one array caps the group at N callers **total across all symbols combined**, not N per symbol — the highest-ranked symbols starve every other symbol of caller slots entirely. `tryPersistentBlast` did exactly this (`callers.sort((a,b)=>b.rank-a.rank); callers.slice(0, MAX_CALLERS_PER_SYMBOL)` on the flat merged array). On a self-review PR with 77 changed symbols, ~57 showed 0 callers even though real callers existed — the top-20-ranked callers (from a handful of symbols) consumed the entire cap. Fix: group by `viaSymbol` into a `Map<string, BlastCallerRow[]>` first, sort+slice `MAX_CALLERS_PER_SYMBOL` **within each group**, then flatten. A constant named `MAX_CALLERS_PER_SYMBOL` does not guarantee the slice is actually scoped per-symbol — verify the grouping happens before the slice. (`src/modules/repo-intel/service.ts:tryPersistentBlast`)

- 2026-07-02: No existing test exercised `tryPersistentBlast`/`getBlastRadius` with more than one changed symbol producing callers at once — a single-group fixture cannot distinguish a correct per-group cap from an incorrect global cap (they produce identical output when there's only one group). This is why the bug in the entry above shipped invisibly through `repo-intel-facade-degraded.test.ts` and `repo-intel-rank-map.test.ts`. When testing any "top-N per group" aggregation, always include ≥2 groups where a naive global slice would visibly differ from the correct per-group result.

- 2026-07-02: **SUPERSEDES the two entries above — they were real bugs but NOT the blocking one.** `src/adapters/depgraph/index.ts`'s `toRel()` normalizes a cruise-returned path via `relative(root, abs)` from `node:path` — on Windows this returns **backslash-separated** paths (`client\src\app\...`). `buildEdges`'s `fileSet.has(from)`/`fileSet.has(to)` membership check then compares against `fileSet` built from `walk.files`, which `pipeline/walk.ts:119` explicitly POSIX-normalizes (`relative(root, full).split(sep).join('/')`). On Windows, `"client\\src\\..." !== "client/src/..."` as Set keys, so **every single edge silently fails the membership check** — `buildEdges` always returns `[]`, `file_edges` stays empty, `resolveReferences`'s join (`file_edges` ⋈ `symbols`) can never produce a candidate, so `references.decl_file` stays NULL for 100% of rows, and `getResolvedCallers`'s `inArray(declFile, changedFiles)` filter then matches zero rows — the entire blast-radius caller pipeline silently degrades to 0 callers for every symbol, on every PR, with `degraded: false` (looks like a clean success). Confirmed empirically on the self-hosted repo: `file_edges` had 0 rows despite 6416 references and 312 indexed files; a standalone repro calling `cruise()` directly found 462 real local edges, but 0 survived the `fileSet` check due to the separator mismatch. No exception is ever thrown (`DepCruiseGraph.buildEdges`'s outer `try/catch` isn't even reached — the mismatch happens after `cruise()` succeeds), so `stats.graphFailed` is never set either — the index run reports itself `status: 'full'` and looks completely healthy. Fix: `toRel()` must also `.split(sep).join('/')`, exactly like `walk.ts`. **This class of bug (POSIX-normalize one side of a Set/Map membership check, forget the other) has zero test coverage anywhere in `repo-intel` — every existing test stubs `depgraph.buildEdges` (`async () => []`) instead of exercising the real adapter.** Added `src/adapters/depgraph/index.test.ts` — a real (unstubbed) `cruise()` call against a same-drive temp fixture, asserting returned edges are POSIX and match a POSIX-normalized `fileSet` — to guard this specifically. Verified live after fix + resync: PR #7 on this repo went from 0/90 downstream entries with callers to 42/90. (`src/adapters/depgraph/index.ts:toRel`)

- 2026-07-02: `dependency-cruiser`'s `cruise()` mis-resolves absolute paths when the analyzed files live on a **different drive letter** than `process.cwd()` on Windows (e.g. fixture under `os.tmpdir()` = `C:\Users\...` while the project/server runs from `D:\...`) — it silently produces 0 modules/dependencies (or throws `ENOENT` while concatenating the cwd's drive with the target's full path, e.g. `D:\...\C:\Users\...`) instead of resolving the absolute path as-is. Not a bug in this codebase, but a real test-authoring trap: any test that calls `DepCruiseGraph.buildEdges`/`cruise()` against a temp fixture must create that fixture on the **same drive** as `process.cwd()` (e.g. `mkdtemp(join(process.cwd(), 'tmp-...'))`), not via `os.tmpdir()`. Production is unaffected — `repo.clonePath` is always under `server/clones/...`, same drive as the server process. (`src/adapters/depgraph/index.test.ts`)

- 2026-07-02: **Third blast-radius bug, found after the depgraph fix above still left ~half of symbols at 0 callers.** `resolveReferences`'s candidate JOIN was strictly 1-hop through `file_edges` (`from_path`'s DIRECT edges only). This codebase's dominant component-import pattern is a re-export barrel — `_components/<Name>/index.ts` doing `export { X } from './X'` — and a barrel file declares ZERO symbols of its own (confirmed: `select * from symbols where path like '%index.ts'` for several barrels returned 0 rows). So `caller.tsx → _components/Name/index.ts` (1 hop) never joins to `symbols` (which has `X` declared in `Name.tsx`, one hop further) — EVERY reference reaching a symbol through a barrel resolved to `decl_file = NULL`, i.e. 0 callers, even though the real call site plainly exists (verified: `page.tsx` imports+renders `DiffTab`/`FindingsTab`/`PrDetailHeader` directly, all 0 callers before the fix). Fix: `resolveReferences` now walks `file_edges` transitively up to `BFS_DEPTH` (2) hops via a recursive CTE before joining to `symbols`. **Naive fix pitfall:** pooling candidates across ALL depths and requiring global uniqueness regressed a *different* set of previously-correct depth-1 resolutions — e.g. a bare function `deleteReview` in `review.repo.ts` AND the dual-emitted bare-name `ReviewRepository.deleteReview` method are both reachable within 2 hops from the same caller, so pooling made that reference newly ambiguous (2 candidates → NULL) even though the depth-1 match alone was already unique. Correct fix: **nearest-depth-wins** — only candidates at the shallowest depth that has ANY match are considered for uniqueness (a `MIN(depth)` CTE step before the uniqueness check), so a depth-1-unique match is never displaced by a coincidental deeper one. Net result on PR #7 (this repo, 90 downstream entries): 0 with callers → 42 (depgraph fix only) → 42 with a *different* 9 gained/9 lost (pooled-depth, regression) → 51 with zero regression (nearest-depth-wins). `ref_with_declfile` (of 10240 total references): 0 → 943 → 1024 → 1069. Remaining 0-caller entries were spot-checked and are legitimate: same-file-only helpers already correctly excluded by the decl-file self-reference fix (e.g. `formatWhen`, privately declared+used inside `ReviewRunAccordion.tsx`), and genuinely-unused exports (e.g. `useGenerateRisks` — declared+exported in `lib/hooks/brief.ts`, zero call sites anywhere in `client/src`). Depth-2 reach is still a documented precision/recall tradeoff (a real ambiguous tie 2 hops out still yields NULL) — that is the intended "honest unresolved, never guess" behavior, not a bug. (`src/modules/repo-intel/repository.ts:resolveReferences`)

## Codebase Patterns

- 2026-07-02: **Product decision (user-confirmed):** `BlastService`'s `groupBlast()` drops any changed symbol with 0 external callers from `downstream` entirely — it does NOT render as an empty/collapsed row. Rationale from the user: "the card shows blast radius only" — a symbol with zero external callers (self-file-only usage, already excluded upstream, or genuinely unused) has no blast radius to report, so listing it as noise defeats the card's purpose. `changed_symbols` (the full, unfiltered list) is kept as-is and still drives the `noDownstream` empty-state message's count (`t("noDownstream", { count: changed_symbols.length })`), so "N functions changed, 0 have external callers" is still shown when everything gets filtered — the total isn't lost, just not rendered as dead rows. Client's "symbols" stat counter was changed from `changed_symbols.length` to `downstream.length` to stay consistent with what's actually in the tree (header count must equal visible row count). Do not revert to rendering all `changed_symbols` unconditionally — that was explicitly rejected after the first three backend fixes made 0-caller rows the common case for legitimate reasons (self-only helpers, dead exports), not just bugs. (`src/modules/blast/service.ts:groupBlast`, `client/.../BlastRadiusCard.tsx`)

- 2026-06-26: `pr_intent` table (`src/db/schema/reviews.ts:48-55`), `Intent`/`PrIntentRecord` Zod contracts (`src/vendor/shared/contracts/brief.ts:9-14`), `upsertIntent`/`getIntent` on `ReviewRepository`, and `review_intent` feature model entry in `platform.ts` all pre-existed before the Intent Layer feature. No migration, no new contract, and no new repo method was needed — only generation logic, routes, and UI were missing. Grep before adding any intent infrastructure. (`src/modules/reviews/repository.ts`, `src/vendor/shared/contracts/brief.ts`)

- 2026-06-22: The `conventions` table, `ConventionCandidate` Zod schema, and `FeatureModelId: 'conventions'` feature model config were all pre-built before the Conventions Extractor feature was implemented. If adding convention-related work, the schema (`src/db/schema/knowledge.ts`), shared contract (`src/vendor/shared/contracts/knowledge.ts`), and feature model entry (`src/modules/settings/feature-models.ts`) already exist — no new migration or contract needed. (`src/db/schema/knowledge.ts`, `src/vendor/shared/contracts/knowledge.ts`)

- 2026-06-22: To resolve a feature model to `{ model, provider }` from workspace settings, use `resolveFeatureModel(container, workspaceId, featureModelId)` — this handles workspace override → registry default. Then pass the provider to `container.llm(provider)` to get the `LLMProvider` adapter. Pattern used in `ConventionsService.extract()`. (`src/modules/conventions/service.ts`)

- 2026-06-22: `service.get()` for skills does NOT automatically include stats — `statsForSkills()` must be called explicitly with the single id, just like `list()` does for the batch. If the skill DTO returned by `GET /skills/:id` shows `agent_count/pull_pct/accept_pct` as `null`, the fix is to add `statsForSkills([row.id])` inside `get()`. Forgetting this is silent — no type error, the fields are just nullish. (`src/modules/skills/service.ts:get`)

- 2026-06-22: Multi-query aggregate pattern in repository: `statsForSkills()` runs 3 separate Drizzle queries (agent count, PR pull count, finding accept count) then merges them into a single `Map<skillId, SkillStats>` via JS. This avoids a complex multi-join SQL query and is efficient when the id list is reasonable (skills per workspace). Follow the same pattern for any new per-skill aggregates: one focused query per metric, merge in JS. (`src/modules/skills/repository.ts:statsForSkills`)

- 2026-06-20: `ReviewRepository` (src/modules/reviews/repository.ts) is a thin wrapper class over the function-level repos in `repository/run.repo.ts`, `review.repo.ts`, `pull.repo.ts`. When adding a new parameter to a repo function (e.g. `completeAgentRun`), the class method signature in `repository.ts` must be updated separately — it does NOT auto-derive from the underlying function's type. (`src/modules/reviews/repository.ts:151`)
- 2026-06-20: Cost calculation (`estimateCost`) lives in `src/adapters/llm/pricing.ts` and is used by the LLM adapters at call time. The `ReviewOutcome` returned by `reviewPullRequest()` already contains a computed `costUsd` — no re-computation needed at persistence time. (`reviewer-core/src/review/run.ts`)
- 2026-06-20: The PR list handler (`src/modules/pulls/routes.ts`) computes derived per-PR aggregates (score, cost) with a separate IN-query + JS Map pattern after the main PR rows query — not via SQL JOIN. When adding a new per-PR aggregate to the list, follow this pattern (see `latestRunCostByPr`). (`src/modules/pulls/routes.ts:119`)

## Tool & Library Notes

- 2026-06-20: `pnpm db:generate` diffs the Drizzle schema files against the migration journal (no live DB needed) and produces a new `.sql` file in `src/db/migrations/`. Never hand-write or edit migration files. Always run `pnpm db:migrate` after generating. (`src/db/migrations/`)

- 2026-06-22: `fflate` exports `zipSync` and `strToU8` alongside `unzipSync`/`strFromU8` — use them in unit tests to create in-process ZIP buffers without touching the filesystem. Pattern: `Buffer.from(zipSync({ 'dir/SKILL.md': strToU8(content) }))`. No extra import needed — they're in the same package already used by the service. (`src/modules/skills/import.service.test.ts`)

- 2026-06-26: Agent `skills:` frontmatter entries are loaded into context at startup (eager) — a 600-line skill costs ~4,500 tokens per invocation regardless of whether the agent uses it. Prefer loading large, infrequently-needed skills lazily via the `Skill` tool. `react-testing-library` (604 lines) was removed from `planner.md`'s eager list: the planner references the skill by name in task definitions, it never writes RTL syntax. (`.claude/agents/planner.md`)

- 2026-07-02: **Repo-wide, not server-specific — noted here since that's where it was hit.** The `pr-self-review` `PreToolUse` hook (`.claude/skills/pr-self-review/scripts/check-gate.sh`) blocks `git push`/`gh pr create`/`gh pr merge` unless `.pr-self-review.json` has a fresh PASS for the current diff hash. An agent cannot self-certify: writing that PASS record for a diff the agent itself authored is blocked by the auto-mode permission classifier as self-approval (bypasses the two-party-review intent of the gate), regardless of how thoroughly the agent actually verified the change (typecheck/tests green, manual diff read). The documented escape hatch (`PR_SELF_REVIEW_OVERRIDE="reason" git push ...`) also does NOT work when set inline as a prefix in an agent's Bash tool command string — `check-gate.sh` runs as a separate process that reads its OWN environment, not the text of the command being gated, and Bash tool shell state (including `export`) does not persist between separate tool calls either. The override only works if set in the actual invoking shell/session environment (i.e. by the human directly, or via persistent harness/OS-level config) — it is not achievable from inside a single agent Bash call. For a low-risk change an agent wants to push on the human's behalf, the only clean paths are: hand the actual `git push`/`gh pr create` back to the human, or (if just demonstrating behavior, not shipping) a DB-only/non-git demonstration that never invokes those commands. (`.claude/skills/pr-self-review/scripts/check-gate.sh`)

## Recurring Errors & Fixes

- 2026-06-22: LLM-supplied file paths (e.g. `evidencePath` from extraction JSON) must be sanitized before any `fs` call — use `path.resolve(fullPath).startsWith(path.resolve(repoPath))` and skip paths that escape the repo root. Same pattern already used in `src/modules/skills/import.service.ts:39`. Without the guard, a prompt-injected model output could cause reads outside the cloned repo directory. (`src/modules/conventions/extractor.ts:verifyEvidence`)

- 2026-06-22: LLMs may return `confidence: 85` (integer) instead of `0.85` (fraction). Clamp immediately after parsing: `Math.max(0, Math.min(1, Number(raw.confidence) || 0))`. Also widen the type check to accept `'string'` for confidence, since some models quote numeric values. Without clamping, the value persists in DB and renders incorrectly (e.g. 9500% confidence bar). (`src/modules/conventions/extractor.ts:callLLM`)

- 2026-06-22: Using `if (!content)` to guard a `Map.get()` result silently drops entries whose value is an empty string (`''`). The correct guard is `if (content === undefined)` — Map.get() returns `undefined` only when the key is absent, not when the value is an empty string. Caught in `assembleSkill` during code review. (`src/modules/skills/import.service.ts:assembleSkill`)

- 2026-06-22: `github.com/.../blob/...` URLs return HTML (the GitHub web UI page), not the raw file. `SkillsImportService.previewFromUrl()` now auto-converts blob URLs to `raw.githubusercontent.com` and rejects responses with `content-type: text/html`. If adding new URL-fetching logic, always validate `content-type` before parsing as text. (`src/modules/skills/import.service.ts:normalizeGitHubUrl`)

- 2026-07-02: Fixed blast-radius `changedSymbols` including `interface`/`type` declarations (e.g. `FooProps`) as top-level entries — these can never have callers (they're never invoked, only referenced as types), so they rendered as permanently-empty accordion rows in `BlastRadiusCard` and inflated the symbol count (77 vs. ~10 real functions on a self-review PR). Fix: filter to `kind ∈ {function, method, class}` in both `getBlastRadius` (ripgrep path) and `tryPersistentBlast` (persistent path) — this filter already existed in `getCallerSignatures` (same file, same reasoning: "type/interface aliases have no call sites") but was never applied to the blast-radius paths. (`src/modules/repo-intel/service.ts:220-260,326-340`)

- 2026-07-02: `tryPersistentBlast`'s `getResolvedCallers` query counted a symbol's own declaration file as one of its "callers" — no `fromPath != declFile` filter existed. The ripgrep fallback path (`getBlastRadius`) already excludes this case (`if (r.fromPath === sym.file) continue`) but the persistent path had no equivalent, so the two code paths disagreed on what counts as a caller. Fix: add `ne(t.references.fromPath, t.references.declFile)` to the `getResolvedCallers` WHERE clause. (`src/modules/repo-intel/repository.ts:getResolvedCallers`)

- 2026-07-02: `blast/service.ts`'s `groupBlast()` originally produced one `downstream` entry per element of `changedSymbols`, not per unique symbol NAME. `repoIntel.getBlastRadius`'s `changedSymbols` is deduped by `(name, file)`, so the same function name declared in two different changed files (e.g. re-declared in two modules) yields two separate `changedSymbols` entries — and since `groupBlast` matches callers by name only (`c.viaSymbol === changedSymbol.name`), both resulting `downstream` rows carried IDENTICAL caller/endpoint/cron data. Symptom in the browser: React "Encountered two children with the same key" console error from `BlastRadiusCard`'s `key={d.symbol}`, plus the same data silently rendered twice. Fix: track a `Set<string>` of seen symbol names and skip `changedSymbols` entries whose name was already emitted, before building each `downstream` push — `changed_symbols` (the separate, unfiltered top-level list) is left as-is since each entry there still carries a distinct `file`. (`src/modules/blast/service.ts:groupBlast`)
