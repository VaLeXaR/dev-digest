- 2026-07-12: **Wired the previously-dead Smart Diff per-file "summary" button/text to a real generate action.** `GroupSection` (`SmartDiffViewer.tsx`) already had `summaryVisible` toggle state and the render logic — it was just permanently unreachable because `file.pseudocode_summary` was always `null` from the server. ## What Doesn't Work

- 2026-07-09: Running the full `cd client && pnpm test` (`vitest run`, no filter) while a **parallel implementer is actively mid-write on a sibling task's file** can hang indefinitely with zero output (confirmed: 2min/3min/10min timeouts all produced no completion, no error) — vitest's transform step appears to stall on a file another process has open/being edited concurrently (observed during T-12 while T-13 was mid-edit on `SkillEditor.tsx`; an isolated `vitest run "src/app/skills/[id]/_components/SkillEditor"` also hung with zero output in the same window). This is not caused by the tests in your own Owned paths. To verify your own task without blocking on the full suite: run `pnpm exec vitest run <your-dir-or-file-glob>` scoped to just your changed test file(s) (NOT `pnpm test -- <filter>` — pnpm forwards a literal `"--"` arg to vitest, which is swallowed and does not actually filter, silently running the whole suite instead). Re-run the full `pnpm test` once sibling implementers have finished, rather than treating a hang as your own regression. in the same synchronous tick are unsafe in Next.js 15 / React 19 concurrent mode** — the scheduler can cancel the first navigation before it commits, leaving the URL unchanged. Pattern seen in `ImportSkillModal.handleConfirm`: calling `onClose()` (which did `router.replace("/skills")`) immediately followed by `router.push("/skills/${id}")` caused the modal to stay open even after a successful import. Fix: use a single `router.replace(destination)` per action, not two consecutive calls. (`src/app/skills/_components/SkillsListView/_components/ImportSkillModal/ImportSkillModal.tsx:51`)

- 2026-06-22: Native `<select>` dropdown is not styleable in dark mode — `color-scheme: dark` and all CSS custom properties are ignored for the browser-native open dropdown list. This affects both light text on white background (low contrast) and colored options. Only fix: replace with a fully custom component. A reusable `Select<T>` with portal rendering exists at `src/vendor/ui/kit/Select.tsx` — use it instead of any native `<select>`. (`src/vendor/ui/kit/Select.tsx`)

- 2026-06-23: `extractMutation` in `ConventionsView.handleRescan` has no `onError` handler — if the POST `/repos/:id/conventions/extract` fails (LLM model not configured, bad API key, network error), the spinner stops and the page silently returns to "No conventions found yet" with zero user feedback. The `onSuccess(data.length === 0)` toast only fires on success. Fix: add `onError: (err) => toast.error(...)` to the `mutate` call. (`src/app/conventions/_components/ConventionsView/ConventionsView.tsx:74-81`)

- 2026-07-02: `SmartDiffViewer` renders unified-diff HUNKS (a few lines of context around each change), not the full file — a `targetLine` requested via click-to-line navigation can legitimately fall outside every rendered hunk (a blast-radius caller's line can be anywhere in the file, not just near a change). A `MutationObserver`-based "wait for `[data-line-no=X]` to mount, then scroll" effect will wait forever for a line that structurally can never appear: the file accordion opens (a sibling effect expands it independently) but nothing scrolls, with zero visible feedback or error. Fix: scroll to the FILE element immediately as a fallback (`[data-file-path]`, always present once expanded), then upgrade to the exact line if/when it mounts — never leave the effect waiting silently with no visible outcome. (`src/app/repos/[repoId]/pulls/[number]/_components/SmartDiffViewer/SmartDiffViewer.tsx`)

- 2026-07-05: **TanStack Query — nulling out query-key arguments to "disable" a `useQuery` call drops its already-fetched `data` the instant the gating condition flips back to false**, because the null args change the query's cache key, not just its `enabled` flag. Pattern that broke: `useLineContext(prId, missingActive ? targetFile : null, missingActive ? targetLine : null, missingActive)` — the moment a fetch succeeded and mounted its result (a `[data-line-no]` element), the SAME scroll effect that watches for that element found it and called `setLineMissing(null)`, flipping `missingActive` to `false` on the next render. That swapped the query's args to `(prId, null, null, false)` → a DIFFERENT `queryKey` than the one the data was cached under → `data` reverted to `undefined` → the just-rendered block unmounted itself one tick after appearing. Fix: keep the query's identifying args (`targetFile`, `targetLine`) STABLE across the enabled/disabled toggle; pass only the boolean through the dedicated `enabled` parameter. `enabled` should gate fetching, never the cache key. (`client/src/app/repos/[repoId]/pulls/[number]/_components/SmartDiffViewer/SmartDiffViewer.tsx`, `client/src/lib/hooks/smart-diff.ts:useLineContext`)
