# Workflow Insights

Process-level retrospective log for multi-agent SDD/run-plan runs — friction, near-misses,
and process recommendations. Companion to per-module `INSIGHTS.md` (codebase knowledge)
and to `docs/retros/ledger.md` (the quantitative trend row for the same runs — this file
is the qualitative "why", that file is the "how much").

> Entries are LLM-generated inference from a single run's transcript and self-reported
> `Process notes`, not independently measured ground truth — a human spot-check is
> expected, same as `INSIGHTS.md`.

## Process Recommendations

- 2026-07-09: On a `failed` task-notification from an async agent, check whether the cause is a stall/API-error (as opposed to a genuine task-level failure) and attempt one `SendMessage` resume before re-dispatching from scratch — this run's resume succeeded and avoided re-incurring ~$10 and 35 minutes. (orchestrator behavior)
- 2026-07-09: Standardize future `researcher` dispatch prompts on the literal `output: compact-digest` directive from Pattern 1 (`.claude/agents/README.md:530`) instead of ad hoc "under N words" phrasing, so digest-sizing is auditable against the documented pattern. (dispatch-prompt convention)
- 2026-07-09: During `grilling`, dispatch `researcher` subagent(s) for codebase-evidence questions per the skill's own instruction rather than defaulting to direct `Grep`/`Read`/`Bash` — the skill currently has no proportionality carve-out for a trivial single-fact check, so either follow it consistently or propose (as a separate, explicitly-approved change) adding an explicit threshold to `grilling/SKILL.md` for when a direct check is acceptable. (`grilling` skill adherence)
- 2026-07-09: After `implementation-planner` returns a `**Next step:**` directive naming a mandatory handoff (e.g. run `grilling`), invoke it immediately rather than pausing to ask the requester for permission — `CLAUDE.md`'s "Mandatory handoffs — do not skip, even mid-conversation" already pre-authorizes the action; asking first only adds an avoidable round-trip for something that isn't actually discretionary. (orchestrator behavior)
- 2026-07-09: `run-plan`'s Stage 1 dispatch model gates on whole-phase completion ("A phase is complete only when every task in it reports DONE... Do not start a dependent phase early"), but within a single phase containing multiple parallel tasks, a task that depends on only ONE sibling (per its own `Depends-on` line) should be dispatched the moment that specific sibling clears — not held until every other sibling in the same phase also reports back. On the `project-context` build this cost ~3.8h of wall-clock (T-05 depended only on T-04, but wasn't dispatched until T-06 and T-07 — both stalled on an unrelated API limit — also finished). Track per-task dependency satisfaction within a phase, not phase-batch completion, when a same-phase task's `Depends-on` doesn't cover every task dispatched alongside it. (`run-plan` skill, Stage 1)
- 2026-07-09: `implementer` agents should be told explicitly not to launch their own verification command with `run_in_background` and then end their turn expecting a notification — subagents don't receive the orchestrator's task-notification mechanism, so that pattern stalls indefinitely until the coordinator notices and intervenes (observed twice in one dispatch, see Friction Points). Add to the implementer definition: run verification synchronously, or if a long-running command must be backgrounded, actively poll and read its result before ending the turn. (`implementer` agent definition)
- 2026-07-10: When a completed task's own Process notes flag a stale or incorrect claim in the plan text and identify the verified-correct alternative, apply that correction directly in the next dependent task's dispatch brief — don't default to "stay consistent with the sibling task's existing (stale) choice." Consistency with a known-wrong pattern isn't a virtue; it just relocates the fix to a later, more expensive stage (see the matching Near-Miss above). (orchestrator behavior, `run-plan` Stage 1)
- 2026-07-10: `architecture-reviewer` cannot run `git diff`/`git status` itself (Read/Glob/Grep/Skill tools only, by design) and self-reported this as a scope gap in its round-1 Process notes on the `onboarding-generator` run — it fell back to trusting the coordinator's hand-typed file list as the diff boundary. Paste actual `git status --short` (or `git diff --stat`) output into future `architecture-reviewer` dispatch prompts instead of a manually enumerated file list, to close this self-reported gap at near-zero cost. (`architecture-reviewer` dispatch-prompt convention)
- 2026-07-10: A single `implementer` task that bundles "build a full page + N section components + i18n + tests + live-browser design verification" (T-08 on `onboarding-generator`: 272K tokens, 168 tool calls, 38.2 minutes, 2 nested verification agents) is disproportionately larger than every sibling task in the same plan and became the run's entire DAG critical path almost by itself (2292.9s of a 3907.2s total wall-clock). Future plans should flag a task of this shape as expected-long upfront, or split "build" from "design-verify" into two dispatches, so a legitimately long-running dispatch doesn't read as a possible hang mid-run. (`implementation-planner` task-sizing guidance)
- 2026-07-11: Add a documented "Researcher → Spec Creator" chaining pattern to `.claude/agents/README.md` (mirroring Pattern 1's Researcher → Implementation Planner digest handoff): before a `spec-creator` dispatch that needs internal-codebase grounding (existing contracts, tables, feature-model ids, sibling components), dispatch a `researcher` subagent with `output: compact-digest` instead of the coordinator performing the exploration itself with raw `Read`/`Grep`/`Glob`/`Bash` calls. Keeps the upside observed this run (rich grounding → zero clarifying round-trips, see What Worked Well) while moving the exploration cost off the coordinator's own context and onto a subagent whose output `spec-creator` still independently re-verifies per its own Hard Rules. (process gap, `.claude/agents/README.md` chaining patterns)
